# simplicial-latent-geometry

**simplicial-latent-geometry** is a Lean 4 formalization project about **testing for latent geometry in random simplicial complexes**. The repository extends the intuition behind graph-geometry detection into the simplicial setting: given an observed complex on many vertices, can we distinguish a purely random null model from a geometric complex generated by points living in an underlying latent space?

At a high level, the project asks when higher-order combinatorial structure contains enough signal to reveal hidden geometry. The repository formalizes that question in Lean by encoding the relevant probabilistic models, parameter matching, volume expressions, and the variance-based detection argument.

| Repository focus | Description |
|---|---|
| Research domain | Random simplicial complexes, geometric inference, probabilistic combinatorics |
| Central question | Can one distinguish a null 2-parameter complex from geometric Čech/Rips alternatives? |
| Main mathematical theme | Detection thresholds and signal-to-noise behavior as ambient dimension changes |
| Technical stack | Lean 4, Mathlib, theorem-spec-driven formalization |
| Portfolio value | Shows formalization of a modern probabilistic research problem rather than only classical textbook results |

## Research problem

The null model in the project is a **2-parameter complex (2PC)**, where edges appear independently with probability \(p\) and 3-cycles are filled independently with probability \(q\). The geometric alternatives are **Čech** or **Rips** complexes built from random points sampled on spaces such as the flat torus or sphere, where simplices arise from proximity.

The central statistical question is whether the observed simplicial complex contains enough higher-order structure to tell these models apart. Put differently, when does geometry leave a detectable signature once the first-order edge density has already been matched?

| Model | Informal description |
|---|---|
| Null 2PC model | A non-geometric random simplicial complex with independently sampled edges and filled triangles |
| Geometric Čech/Rips model | A simplicial complex induced by proximity among random latent-space points |
| Key observable | Filled triangle statistics and their fluctuation behavior |
| Detection mechanism | Compare expectation and variance under null versus geometric alternatives |

## What this repository formalizes

The repository is organized around the ingredients needed to turn that research question into a mechanically checkable mathematical argument. The most important pieces are the formal definitions of the competing models, the parameter matching that makes the comparison fair, and the proof machinery needed to show that the geometric model has a qualitatively different variance scale.

| Formalization target | Why it matters |
|---|---|
| 2PC and Čech model definitions | Establishes the exact probabilistic objects being compared |
| Parameter matching formulas | Ensures the null and geometric models are aligned at the level of low-order observables |
| Volume integrals such as `V_f` and `V_e` | Capture the geometric quantities driving filled-triangle behavior |
| Expectation and variance of filled triangle counts | Supplies the statistic used for detection |
| Signal-to-noise argument | Encodes the actual detection threshold logic |

## Core mathematical story

The high-level narrative is that geometry becomes statistically visible through higher-order dependence. Even if a null model is tuned to match lower-order quantities such as edge density, the geometric model induces structured correlations that change the behavior of filled triangle counts. In particular, the project is built around the claim that the geometric model and the null model differ in variance scale, yielding a usable signal-to-noise separation.

That makes the repository interesting both mathematically and from a portfolio perspective. It is not just another Lean exercise; it is a formalization of a contemporary research-style argument in probabilistic geometry.

## Development workflow

The working loop is paper-driven. Mathematical notes and literature synthesis are used to define theorem targets, those targets are translated into Lean files with structured proof intent, and the proofs are then iteratively completed and refined.

| Step | Typical command |
